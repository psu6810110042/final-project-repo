## 1. วัตถุประสงค์ของการจัดทำ Statement of Work

เอกสารฉบับนี้จัดทำขึ้นเพื่อกำหนดขอบเขตของโครงการ (Scope of Work) โดยระบุข้อกำหนดและรายละเอียดที่สำคัญสำหรับการพัฒนาระบบร้านขายยาออนไลน์ เพื่อใช้เป็นข้อตกลงร่วมกันระหว่างผู้ว่าจ้างและผู้พัฒนา และใช้เป็นแนวทางในการวางแผน บริหารจัดการ และติดตามความคืบหน้าของโครงการ เพื่อให้มั่นใจว่าผลลัพธ์ที่ได้จะสอดคล้องกับความต้องการและวัตถุประสงค์ของโครงการอย่างแท้จริง

## 2. วัตถุประสงค์ของโครงการ

2.1 เพื่อพัฒนาระบบ E-Commerce สำหรับจำหน่ายยาและเวชภัณฑ์ออนไลน์ (คล้ายรูปแบบร้าน Watsons) ที่มีความถูกต้อง ปลอดภัย และเป็นไปตามมาตรฐาน
2.2 เพื่ออำนวยความสะดวกให้กับลูกค้าในการสั่งซื้อยา โดยมีระบบคัดกรองความปลอดภัยสำหรับการสั่งซื้อยาอันตรายและยาควบคุม
2.3 เพื่อเพิ่มประสิทธิภาพในการบริหารจัดการคลังสินค้า (Inventory Management) ด้วยระบบ FEFO (First Expired, First Out) และการแจ้งเตือนสินค้าใกล้หมดอายุ
2.4 เพื่อสนับสนุนการทำงานของเภสัชกรในการตรวจสอบใบสั่งยา (Prescription Verification) และอนุมัติการจำหน่ายยาตามกฎหมาย

## 3. ขอบเขตของโครงการ

ขอบเขตของการพัฒนาระบบร้านขายยาออนไลน์ มีรายละเอียดดังต่อไปนี้

### 3.1 ขอบเขตของการดำเนินงาน (Scope of Work)

การดำเนินงานแบ่งออกเป็น 3 ส่วนหลัก ได้แก่ ส่วน Frontend (ลูกค้า), ส่วน Backend (ผู้ดูแลระบบ/เภสัชกร), และส่วน Database (ฐานข้อมูล)

#### **A. ส่วนลูกค้า (Customer & Public Interface)**

พัฒนาด้วย **React** และ **NextJS** โดยมีฟีเจอร์หลักดังนี้:

**1. ระบบยืนยันตัวตนและความปลอดภัย (Authentication & Security)**
*   **Register/Login**: รองรับการลงทะเบียนและเข้าสู่ระบบด้วย Email หรือ เบอร์โทรศัพท์
*   **PDPA Consent**: ระบบขอความยินยอมในการเก็บข้อมูลสุขภาพส่วนบุคคล (PDPA)
*   **User Profile**:
    *   จัดการข้อมูลส่วนตัวและที่อยู่สำหรับจัดส่ง (Address Book)
    *   ระบบบันทึกข้อมูลสุขภาพ: โรคประจำตัว และ ประวัติการแพ้ยา (เพื่อใช้ในการแจ้งเตือนเมื่อซื้อยา)

**2. หน้าร้านค้า (Storefront)**
*   **Search & Filter**: ระบบค้นหาอัจฉริยะ ลิงก์กับชื่อยา หรือ อาการ พร้อมตัวกรองประเภทสินค้า (ยาอันตราย / ยาทั่วไป / เวชภัณฑ์ / อาหารเสริม)
*   **Product List/Grid**: แสดงรายการสินค้าพร้อมรูปภาพ ราคา และป้ายกำกับสถานะ (เช่น "สินค้าหมด", "ยาอันตราย")
*   **Product Detail**: แสดงรายละเอียดเชิงลึก ได้แก่ สรรพคุณ, วิธีใช้, คำเตือน (Box Warning), และวันหมดอายุของล็อตที่ใกล้หมดที่สุด

**3. กระบวนการสั่งซื้อ (Buying Flow)**
*   **Smart Cart**: ตะกร้าสินค้าอัจฉริยะ แยกรายการระหว่าง "สินค้าทั่วไป" (สามารถชำระเงินได้ทันที) และ "ยาควบคุม/ยาอันตราย" (ต้องรอการอนุมัติ)
*   **Prescription Upload**: ฟอร์มอัปโหลดรูปภาพใบสั่งแพทย์ รองรับเฉพาะรายการยาที่ต้องการใบสั่งยา ก่อนเข้าสู่หน้าชำระเงิน
*   **Checkout & Payment**: หน้าสรุปยอดรายการสั่งซื้อ เลือกวิธีการจัดส่ง และฟอร์มแนบหลักฐานการโอนเงิน (Slip Upload)
*   **Order History**: ดูประวัติการสั่งซื้อและสถานะอย่างละเอียด (รออนุมัติยา -> รอชำระเงิน -> รอตรวจสอบยอด -> จัดส่งแล้ว) พร้อมปุ่ม Re-order สำหรับสั่งซื้อซ้ำ

#### **B. ส่วนผู้ดูแลระบบและเภสัชกร (Admin & Pharmacist Dashboard)**

**1. ระบบจัดการสินค้า (Product Management)**
*   **CRUD System**: เครื่องมือสำหรับ เพิ่ม/ลบ/แก้ไข ข้อมูลสินค้า
*   **Batch Management**: ระบบผูกสินค้ากับล็อตการผลิต (Batch) เพื่อติดตามวันหมดอายุและต้นทุนของแต่ละล็อต
*   **Stock Alert View**: หน้าจอ Dashboard แจ้งเตือนรายการ "ยาใกล้หมดอายุ (Near Expiry)" และ "สินค้าคงคลังต่ำ (Low Stock)"

**2. ระบบจัดการคำสั่งซื้อ (Order Management)**
*   **Prescription Verify Console**: หน้าจอสำหรับเภสัชกร เพื่อดูภาพใบสั่งยาและข้อมูลสุขภาพลูกค้า แล้วทำการกด "อนุมัติ" หรือ "ปฏิเสธ" คำสั่งซื้อ
*   **Payment Verify**: หน้าจอตรวจสอบสลิปโอนเงิน และอัปเดตสถานะคำสั่งซื้อเมื่อได้รับการชำระเงินเรียบร้อย

**3. ระบบรายงาน (Reporting)**
*   **Dashboard**: สรุปยอดขายรายวัน / รายเดือน
*   **Export Data**: รองรับการนำข้อมูลคำสั่งซื้อและรายงานสินค้าคงคลังออกมาเป็นไฟล์ Excel (CSV/XLSX)

#### **C. ส่วน Backend และฐานข้อมูล (Business Logic & Database)**

**1. ระบบจัดการสินค้าและคลัง (Inventory Logic)**
*   **FEFO System (First Expired, First Out)**: ระบบตัดสต็อกอัตโนมัติ โดยเลือกตัดจาก Batch ที่กำลังจะ "หมดอายุก่อน" เพื่อลดของเสีย
*   **Stock Reservation**:
    *   ตัดสต็อกแบบชั่วคราว (Reserve) ทันทีที่มีการยืนยันคำสั่งซื้อ
    *   คืนสต็อก (Revert) อัตโนมัติกรณีลูกค้ายกเลิกคำสั่งซื้อ หรือเภสัชกรไม่อนุมัติใบสั่งยา

**2. ระบบควบคุมการสั่งจ่ายยา (Prescription Gate)**
*   **Gate Logic**: หากในตะกร้ามีสินค้าที่ระบุ `is_prescription_required = true` ระบบจะล็อกสถานะห้ามชำระเงิน จนกว่า Admin/Pharmacist จะเปลี่ยนสถานะเป็น `presc_approved`
*   **Status Workflow**: `Pending Review` (รอตรวจยา) -> `Payment Pending` (อนุมัติแล้ว รอโอน) -> `Paid` (โอนแล้ว รอตรวจสลิป) -> `Shipped` (จัดส่ง)

**3. การจัดการข้อมูลและความปลอดภัย (Data Management)**
*   **Encryption**: เข้ารหัสข้อมูลที่มีความอ่อนไหวสูง (Sensitive Data) ก่อนบันทึกลงฐานข้อมูล ได้แก่ ชื่อ-ที่อยู่จริง และ ข้อมูลโรคประจำตัว/การแพ้ยา

**4. การออกแบบฐานข้อมูล (Database Schema Design Concept)**
*   **Users Collection**: เก็บข้อมูล Login, Role (Admin/Customer), ข้อมูลสุขภาพ (Encrypted)
*   **Products Collection**: เก็บข้อมูลยา, เลข อย., คำเตือน, สถานะ Rx (ต้องใช้ใบสั่งยาไหม)
*   **Product_Batches Collection**: เก็บข้อมูล Lot Number, วันหมดอายุ, ต้นทุน, จำนวนคงเหลือ (หัวใจสำคัญของ FEFO)
*   **Orders Collection**: เก็บรายการสั่งซื้อ, สถานะ (Status Enum), ลิงก์รูปใบสั่งยา/สลิป, ID เภสัชกรผู้อนุมัติ

**5. ความต้องการด้านระบบ (System Requirements)**
*   **Responsive Design**: รองรับการแสดงผลได้ดีทั้งบน Mobile และ Desktop
*   **Browser Support**: รองรับ Web Browser มาตรฐานเวอร์ชันปัจจุบัน (Chrome, Safari, Edge)

#### **D. สิ่งที่อยู่นอกเหนือขอบเขตงาน (Out of Scope)**
*   **การจัดหาและลงข้อมูลสินค้าจริง (Data Entry)**: ผู้ว่าจ้างต้องเป็นผู้ดำเนินการเอง
*   **การขอใบอนุญาตทางกฎหมาย**: เกี่ยวกับการขายยาออนไลน์ และการจดทะเบียนพาณิชย์อิเล็กทรอนิกส์
*   **ระบบชำระเงินผ่านบัตรเครดิต (Payment Gateway)**: (ในขอบเขตงานรองรับเพียงการแนบสลิปโอนเงิน หากต้องการเชื่อมต่อต้องมีข้อตกลงเพิ่มเติม)
*   **ค่าใช้จ่ายด้าน Infrastructure**: ค่า Server, Domain Name และ SMS Gateway (ผู้ว่าจ้างเป็นผู้รับผิดชอบค่าใช้จ่าย)

### 3.2 ระยะเวลาในการดำเนินโครงการ

ระยะเวลาดำเนินโครงการรวมทั้งสิ้นประมาณ [X] เดือน (ระบุตามจริง) โดยแบ่งออกเป็นระยะต่างๆ ดังนี้:
*   [วว/ดด/ปป] - เริ่มต้นโครงการ
*   [วว/ดด/ปป] - สิ้นสุดโครงการ

### 3.3 งวดงานและงวดเงิน

*(ส่วนนี้สำหรับระบุรายละเอียดการส่งมอบงานและการชำระเงิน หากเป็นการจ้างงานจริง)*
*   **งวดที่ 1**: ส่งมอบเอกสารแผนงาน (Project Plan) และ SRS
*   **งวดที่ 2**: ส่งมอบระบบส่วน Frontend และ Design
*   **งวดที่ 3**: ส่งมอบระบบส่วน Backend, Database และ ระบบเชื่อมต่อ API พร้อมผลการทดสอบ (UAT)
*   **งวดที่ 4**: ส่งมอบ Source Code ต้นฉบับ และ Deploy ขึ้น Server ของผู้ว่าจ้าง พร้อมคู่มือการใช้งาน

### 3.4 การรายงานความคืบหน้าของโครงการ

ผู้บริหารโครงการ (Project Manager) จะจัดส่งรายงานความคืบหน้า (Progress Report) ให้กับผู้ว่าจ้าง [Weekly/Monthly] เพื่อแจ้งสถานะการทำงาน ปัญหา และแนวทางการแก้ไข

### 3.5 ปัจจัยที่มีผลกระทบต่อโครงการ

*   **กฎหมายและข้อบังคับ**: การเปลี่ยนแปลงกฎหมายเกี่ยวกับยาและเวชภัณฑ์ อาจส่งผลต่อ Logic ของระบบการขาย
*   **ความปลอดภัยของข้อมูล**: ความเสี่ยงด้าน Cybersecurity และการปฏิบัติตาม PDPA อย่างเคร่งครัด

## 4. ข้อตกลงร่วมกัน

4.1 ความต้องการระบบจะถูกบริหารจัดการและบันทึกไว้อย่างเป็นลายลักษณ์อักษร
4.2 การติดต่อสื่อสารระหว่างทีมพัฒนาและผู้ว่าจ้างจะทำผ่านช่องทางที่ตกลงกันไว้ (เช่น Email, Meeting) เพื่อให้มีหลักฐานอ้างอิง

## 5. การบริหารการเปลี่ยนแปลง

หากมีการเปลี่ยนแปลงขอบเขตงาน (Change Request) นอกเหนือจากที่ระบุไว้ในเอกสารนี้ จะต้องมีการประเมินผลกระทบด้านเวลาและค่าใช้จ่าย และต้องได้รับความเห็นชอบจากทั้งสองฝ่ายเป็นลายลักษณ์อักษรก่อนเริ่มดำเนินการ
